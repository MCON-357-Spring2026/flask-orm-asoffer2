Flask ORM with SQLAlchemy + Pytest

Goal: Use ORM models (tables/classes), relationships (PK/FK), and sessions (transactions) to build a Flask JSON API — and test it with pytest using the Flask test client.
This lesson reuses earlier DB concepts: tables, constraints, relationships/joins, and commit/rollback.

1) Quick refresher: how ORM maps to database concepts

  • Models represent tables (e.g., Student, Assignment, Grade).
  • Columns represent fields (e.g., email, max_points).
  • Primary keys become id columns marked primary_key=True.
  • Foreign keys become db.ForeignKey(...) columns.
  • Relationships are Python links between tables (e.g., Student.grades, Assignment.grades).
  • Session is how the ORM tracks changes and performs transactions (commit/rollback).
Important: ORM does not remove the need to understand SQL. It automates repetitive work (CRUD + mapping + parameterization), but you still need the underlying DB concepts to debug and design correctly.

2) The app factory pattern (why tests like it)

A common Flask best practice is an app factory: a function called create_app() that builds and returns a Flask app. This is extremely helpful for testing because each test suite can create an app configured for tests. 

# src/flask_orm/app.py
def create_app(test_config=None):
    app = Flask(__name__)
    app.config.from_mapping(
        SQLALCHEMY_DATABASE_URI="sqlite:///data/app.db",
        SQLALCHEMY_TRACK_MODIFICATIONS=False,
    )
    if test_config:
        app.config.update(test_config)

    db.init_app(app)
    app.register_blueprint(api)
    return app
Testing trick: In tests, we override the database URL to use sqlite:///:memory: (an in-memory database) so every test run starts clean and runs fast.

3) ORM relationships + constraints (one grade per student per assignment)

Here’s the key data rule used in this project:

  • One Student can have many Grades
  • One Assignment can have many Grades
  • But a student can only have one grade per assignment
# Grade model has a UNIQUE constraint on (student_id, assignment_id)
__table_args__ = (
    db.UniqueConstraint("student_id", "assignment_id", name="uq_grade_student_assignment"),
)

In your API, that means “posting the same student+assignment grade twice” should return an error (we use HTTP 409 Conflict).

4) Testing Flask routes with pytest + Flask test client

The Flask test client calls your routes without starting a server. Think of it as Postman that runs inside your test code.

Why we test this way:
  • Fast feedback (seconds, not minutes)
  • Repeatable (every student runs the same test suite)
  • Perfect for autograding (GitHub Classroom)
  • Encourages clean design (app factory + dependency injection via config)

4.1 Install and run

pip install -r requirements.txt
pytest -q

4.2 Basic pytest fixtures: app + client

Fixtures are reusable setup functions. We’ll build:

  • app fixture: creates the Flask app configured for tests
  • client fixture: gives us a test client to call routes
import pytest
from src.flask_orm.app import create_app
from src.flask_orm.extensions import db

@pytest.fixture()
def app():
    app = create_app({
        "TESTING": True,
        "SQLALCHEMY_DATABASE_URI": "sqlite:///:memory:",
    })
    with app.app_context():
        db.create_all()           # create tables for this test session
        yield app                 # provide app to tests
        db.session.remove()       # cleanup session state
        db.drop_all()             # drop tables so nothing leaks to other tests

@pytest.fixture()
def client(app):
    return app.test_client()
Common gotcha: Database operations require an app.app_context(). That’s why we create tables inside the context.

5) Route testing patterns (with detailed examples)

5.1 Health check test

def test_health(client):
    res = client.get("/health")
    assert res.status_code == 200
    assert res.get_json() == {"status": "ok"}

5.2 Testing a POST endpoint (create)

For JSON APIs, you send JSON using json={...}. Flask parses it into request.get_json(). 

def test_create_student(client):
    res = client.post("/students", json={"name": "Ava", "email": "ava@example.com"})
    assert res.status_code == 201

    body = res.get_json()
    assert "id" in body
    assert body["name"] == "Ava"
    assert body["email"] == "ava@example.com"

5.3 Testing validation errors (400 Bad Request)

If required fields are missing, your API should respond with 400. 

def test_create_student_missing_fields(client):
    res = client.post("/students", json={"name": "Ava"})
    assert res.status_code == 400
    assert "error" in res.get_json()

5.4 Testing uniqueness constraint behavior (409 Conflict)

def test_duplicate_student_email(client):
    res1 = client.post("/students", json={"name": "Ava", "email": "ava@example.com"})
    assert res1.status_code == 201

    res2 = client.post("/students", json={"name": "Other", "email": "ava@example.com"})
    assert res2.status_code == 409
    assert "error" in res2.get_json()

5.5 Testing GET list endpoints

def test_list_students(client):
    client.post("/students", json={"name": "Ava", "email": "ava@example.com"})
    client.post("/students", json={"name": "Noah", "email": "noah@example.com"})

    res = client.get("/students")
    assert res.status_code == 200
    data = res.get_json()

    assert isinstance(data, list)
    assert len(data) == 2

5.6 Testing GET by id + 404 Not Found

def test_get_student_not_found(client):
    res = client.get("/students/999999")
    assert res.status_code == 404
    assert res.get_json()["error"] == "not found"

5.7 Testing PATCH (update)

def test_update_student_name(client):
    created = client.post("/students", json={"name": "Ava", "email": "ava@example.com"}).get_json()
    sid = created["id"]

    res = client.patch(f"/students/{sid}", json={"name": "Ava Chen"})
    assert res.status_code == 200
    assert res.get_json()["name"] == "Ava Chen"

5.8 Testing DELETE

def test_delete_student(client):
    sid = client.post("/students", json={"name": "Ava", "email": "ava@example.com"}).get_json()["id"]

    res = client.delete(f"/students/{sid}")
    assert res.status_code == 204

    # After deletion, GET should be 404
    res = client.get(f"/students/{sid}")
    assert res.status_code == 404

6) Testing nested routes (grades belong to student + assignment)

6.1 Setup: create assignment first

def test_add_grade_flow(client):
    # Create assignment
    a = client.post("/assignments", json={"title": "HW 1", "max_points": 100}).get_json()
    aid = a["id"]

    # Create student
    s = client.post("/students", json={"name": "Ava", "email": "ava@example.com"}).get_json()
    sid = s["id"]

    # Add grade
    res = client.post(f"/students/{sid}/grades", json={"assignment_id": aid, "score": 95})
    assert res.status_code == 201
    g = res.get_json()
    assert g["student_id"] == sid
    assert g["assignment_id"] == aid
    assert g["score"] == 95

6.2 Duplicate grade (same student + assignment) should fail

def test_duplicate_grade_conflict(client):
    aid = client.post("/assignments", json={"title": "Quiz 1", "max_points": 10}).get_json()["id"]
    sid = client.post("/students", json={"name": "Ava", "email": "ava@example.com"}).get_json()["id"]

    res1 = client.post(f"/students/{sid}/grades", json={"assignment_id": aid, "score": 9})
    assert res1.status_code == 201

    res2 = client.post(f"/students/{sid}/grades", json={"assignment_id": aid, "score": 10})
    assert res2.status_code == 409
    assert "error" in res2.get_json()

6.3 List grades returns assignment info

def test_list_grades_includes_assignment(client):
    a = client.post("/assignments", json={"title": "HW 1", "max_points": 100}).get_json()
    s = client.post("/students", json={"name": "Ava", "email": "ava@example.com"}).get_json()

    client.post(f"/students/{s['id']}/grades", json={"assignment_id": a["id"], "score": 95})

    res = client.get(f"/students/{s['id']}/grades")
    assert res.status_code == 200
    grades = res.get_json()
    assert len(grades) == 1
    assert grades[0]["assignment"]["title"] == "HW 1"

7) Extra pytest tips that help a lot in Flask projects

7.1 Use clear test names and structure

  • Name tests like test_create_student_missing_fields so failure output is self-explanatory.
  • Keep tests short: arrange → act → assert.

7.2 Prefer in-memory databases for unit tests

Use sqlite:///:memory: so each run is isolated and fast.

7.3 If a test fails, print response to debug

res = client.post("/students", json={...})
print(res.status_code, res.get_json())

7.4 Don't forget rollback patterns in app code

If a commit fails (constraint violation), always rollback the session. If you don’t, the session can remain in a broken state and future operations may fail.
