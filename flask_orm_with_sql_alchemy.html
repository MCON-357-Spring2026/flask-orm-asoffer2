Flask ORM with SQLAlchemy: Fast Track

Goal: Use an ORM to model tables as Python classes, manage transactions through a session, and build Flask routes that do CRUD safely and cleanly.
Assumption: you already know primary keys, foreign keys, and joins.

1) ORM mindset (what changes)

Raw SQL mindset ORM mindset
You write SQL strings You work with Python objects + query builders
Manual mapping: row → dict → object Mapping is automatic (model ↔ table)
Transaction control via connection Transaction control via session (commit/rollback)
Important: ORM does not replace database knowledge. It gives you safer, cleaner code and fewer mistakes—especially for CRUD and relationships.

2) Two ways to use SQLAlchemy in Flask

  • Flask-SQLAlchemy (popular Flask extension): easiest for teaching; integrates session and config.
  • Plain SQLAlchemy (SQLAlchemy 2.x style): more explicit; closer to “core” SQLAlchemy patterns.
For this course, Flask-SQLAlchemy is usually the best on-ramp. It’s widely used, reduces boilerplate, and keeps focus on concepts.

3) Setup: packages + config

pip install flask flask-sqlalchemy

Typical Flask config for SQLite (dev):

from flask import Flask
from flask_sqlalchemy import SQLAlchemy

app = Flask(__name__)
app.config["SQLALCHEMY_DATABASE_URI"] = "sqlite:///app.db"
app.config["SQLALCHEMY_TRACK_MODIFICATIONS"] = False

db = SQLAlchemy(app)
SQLite is great for learning and demos. In production you often use PostgreSQL, and the ORM patterns stay the same.

4) Models (tables as classes)

Example: Students and Grades (one-to-many)

from datetime import datetime

class Student(db.Model):
    __tablename__ = "students"

    id = db.Column(db.Integer, primary_key=True)
    name = db.Column(db.String(120), nullable=False)
    email = db.Column(db.String(255), nullable=False, unique=True)

    grades = db.relationship("Grade", back_populates="student", cascade="all, delete-orphan")

class Grade(db.Model):
    __tablename__ = "grades"

    id = db.Column(db.Integer, primary_key=True)
    score = db.Column(db.Integer, nullable=False)
    created_at = db.Column(db.DateTime, nullable=False, default=datetime.utcnow)

    student_id = db.Column(db.Integer, db.ForeignKey("students.id"), nullable=False)
    student = db.relationship("Student", back_populates="grades")

Creating the tables (dev only)

with app.app_context():
    db.create_all()
Reality check: In real teams, schema changes are handled by migrations (Alembic/Flask-Migrate). We’ll mention migrations later, but we can start with create_all() for learning.

5) Sessions: commit, rollback, and why they matter

Flask-SQLAlchemy uses db.session to track new/changed objects. Your changes are not permanent until you commit. 

# Create
student = Student(name="Ava", email="ava@example.com")
db.session.add(student)
db.session.commit()
# Transaction safety pattern
try:
    student.email = "ava2@example.com"
    db.session.commit()
except Exception:
    db.session.rollback()
    raise

6) Querying (ORM “SELECT”)

Get all

students = Student.query.order_by(Student.name).all()

Get one

student = Student.query.filter_by(email="ava@example.com").first()

Get by primary key

student = db.session.get(Student, 1)

Filtering

top = Grade.query.filter(Grade.score >= 90).all()

Join-like queries

rows = db.session.query(Student.name, Grade.score)\
    .join(Grade, Grade.student_id == Student.id)\
    .order_by(Student.name)\
    .all()
Under the hood, SQLAlchemy generates parameterized SQL (safe) and executes it through the driver.

7) Flask routes for CRUD (JSON API style)

Create student (POST /students)

from flask import request, jsonify

@app.post("/students")
def create_student():
    data = request.get_json() or {}
    name = data.get("name")
    email = data.get("email")

    if not name or not email:
        return {"error": "name and email are required"}, 400

    student = Student(name=name, email=email)
    db.session.add(student)

    try:
        db.session.commit()
    except Exception:
        db.session.rollback()
        return {"error": "email must be unique"}, 409

    return {"id": student.id, "name": student.name, "email": student.email}, 201

Read students (GET /students)

@app.get("/students")
def list_students():
    students = Student.query.order_by(Student.name).all()
    return jsonify([{"id": s.id, "name": s.name, "email": s.email} for s in students])

Update student (PATCH /students/<id>)

@app.patch("/students/<int:student_id>")
def update_student(student_id):
    s = db.session.get(Student, student_id)
    if not s:
        return {"error": "not found"}, 404

    data = request.get_json() or {}
    if "name" in data:
        s.name = data["name"]
    if "email" in data:
        s.email = data["email"]

    try:
        db.session.commit()
    except Exception:
        db.session.rollback()
        return {"error": "update failed (possibly duplicate email)"}, 409

    return {"id": s.id, "name": s.name, "email": s.email}

Delete student (DELETE /students/<id>)

@app.delete("/students/<int:student_id>")
def delete_student(student_id):
    s = db.session.get(Student, student_id)
    if not s:
        return {"error": "not found"}, 404

    db.session.delete(s)
    db.session.commit()
    return {}, 204

8) Relationships in routes (nested resources)

Add a grade for a student: POST /students/<id>/grades

@app.post("/students/<int:student_id>/grades")
def add_grade(student_id):
    s = db.session.get(Student, student_id)
    if not s:
        return {"error": "student not found"}, 404

    data = request.get_json() or {}
    score = data.get("score")
    if score is None:
        return {"error": "score is required"}, 400

    g = Grade(score=int(score), student=s)  # relationship assignment
    db.session.add(g)
    db.session.commit()

    return {"id": g.id, "score": g.score, "student_id": s.id}, 201

9) Common mistakes (and fixes)

  • Forgetting app context for create_all() or DB work in scripts → use with app.app_context():
  • Forgetting commit after write operations → changes won’t persist
  • Not rolling back after an exception → session can be stuck “dirty”
  • N+1 queries (lots of small queries) → learn eager loading later
